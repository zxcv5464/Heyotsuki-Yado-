# 薪資分潤 Hotfix 驗收清單

## SQL 與鎖定

- [ ] 依序套用六份薪資 migration。
- [ ] 新建草稿批次、不手動重算，直接鎖定會先產生最新明細。
- [ ] 修改公共池、舞蹈參與者或直接歸屬後直接鎖定，結果採最新資料。
- [ ] 有待分配項目時不能鎖定。
- [ ] 來源可分配總額與系統分配總額不一致時不能鎖定。
- [ ] 重複鎖定不會重複產生明細。
- [ ] locked 批次不能 INSERT、UPDATE、DELETE `payroll_entries`，包含 `manual_adjustment`。
- [ ] draft 批次仍可重算與新增手動調整。

## 舞蹈

- [ ] `祈願之舞 x 1` 建立 1 場並使用訂單單筆金額。
- [ ] `祈願之舞 x 4` 必須建立 4 場。
- [ ] 只建立 2 場時，重算顯示待處理，鎖定前顯示提醒並拒絕鎖定。
- [ ] `320,000 / 4` 為四場各 `80,000`。
- [ ] `100,001 / 3` 依場次序號為 `33,334`、`33,334`、`33,333`。
- [ ] 每場依參與者登記順序平均，尾數依順序分配。
- [ ] 同一場同一員工不能重複；同一員工可參與不同場次。

## 預約排除公共池

- [ ] 當日 `pending`、`confirmed`、`completed` 預約指名員工會被停用並顯示排除原因。
- [ ] 當日 `cancelled`、`no_show` 預約指名員工不排除。
- [ ] 點餐指定員工不影響公共池排除。
- [ ] 舞蹈參與者仍顯示完整員工名單。
- [ ] 預約排除者不出現在公共池可勾選列表，僅顯示精簡摘要。

## 直接歸屬補選

- [ ] 缺少 `selected_staff_id` 的 `direct_staff` 品項會列為待分配。
- [ ] 管理員可在薪資頁選擇受益員工並儲存。
- [ ] 補選後重算，該項目改為直接歸屬，待分配數量下降。
- [ ] 補選資料只寫入 `payroll_source_assignments`，不修改原始訂單品項。
- [ ] locked 批次不能修改補選資料。

## 手動調整與回歸

- [ ] owner / admin 可在草稿批次建立含原因的手動調整。
- [ ] 空白原因與零金額被拒絕。
- [ ] 手動調整顯示於預覽、CSV、文字摘要及員工合計。
- [ ] 批次鎖定後不能新增、修改或刪除既有 `payroll_entries`，包含手動調整。
- [ ] 草稿批次可新增補登舞蹈分潤，且不新增或修改訂單、訂單品項與營業報表資料。
- [ ] 已鎖定批次可新增補登舞蹈分潤，原有 `payroll_entries` 不被修改。
- [ ] 補登金額會依勾選員工順序平均拆分，尾數仍會完整分配。
- [ ] 補登必須填原因、非零金額與至少一位參與者。
- [ ] 補登後不可更新或刪除；更正採新增反向金額補登。
- [ ] 補登顯示於薪資預覽、員工合計、CSV 與文字摘要。
- [ ] 訂單、預約、菜單、營業報表與花牌專案不受影響。
- [ ] `node --check js/admin-payroll.js` 通過。
- [ ] `admin/payroll.html` 沒有重複 ID，所有本地 CSS／JS 引用存在。
